VCP Trading System — project landing page for Solutions Engineer hiring managers

01 — why this exists

Most tools tell you what to trade.
I built one that tells me when, why, and whether I'm ready.

The VCP Trading System is a full-stack stock screening and trade management application — built from scratch — that identifies institutional-quality momentum setups from the S&P 500, scores them against a rigorous 100-point model, and walks me through every decision from initial signal to final exit.

Built Apr – May 2026 Python · Streamlit · SQLite ~8,700 lines · single file Live on localhost
02 — the problem

I was screening stocks manually — a combination of TradingView watch lists, spreadsheets tracking position sizing, and a notes app for pre-trade reasoning. Each tool did one thing adequately and none of them talked to each other.

The real problem was decision lag: by the time I'd pulled price data, checked my sizing rules, reviewed the chart, and written down my reasoning, 20 minutes had passed and the moment had either passed or I'd talked myself into a worse trade. The process was leaky enough that I couldn't tell whether a bad outcome came from a bad setup or a bad decision.

The specific moment that tipped me was realising I had no reliable way to answer "what made this trade worth taking?" after the fact. The data was scattered across three tools and my own memory. I needed one system that captured the full chain — from signal to sizing to outcome — in a form I could actually learn from.

03 — what it does
01
Screener
Runs every S&P 500 stock through ten structural filters, then scores each survivor on a 100-point model. What comes out the other end is a ranked shortlist of setups worth looking at — not a raw list of 500 stocks.
02
Watchlist
Tracks shortlisted stocks in real time — proximity to entry, volume behaviour, pattern health. It flags when a setup is tightening toward the trigger so I'm not manually checking charts every hour.
03
Pre-trade checklist
Before any position is opened, the system calculates risk-reward, position size, and target levels automatically. The checklist enforces that the trade passes a defined standard before I act — removing the "gut feel override" problem.
04
Position tracker
Live P&L, alert engine, and exit tracking per lot. It records market conditions and emotional state at entry — because I want to know eventually whether my worst trades happen on the same day I override my own process.
05
Performance archive
Closed trades with full analytics — win rate, average risk multiple, outcome by market condition. The point isn't to track returns. It's to have enough data to know what to improve next.
04 — how I built it

The stack is Python 3.12, Streamlit for the interface, SQLite for persistence, and yfinance plus a financial data API for market and fundamentals data. Everything lives in a single ~8,700-line file — a deliberate choice for a solo build where simplicity outweighs modularity. Version-controlled on GitHub throughout.

I used Claude Code as the primary development environment and Cursor as an AI-assisted IDE alongside it — not to generate code I didn't understand, but to compress the feedback loop between "I want this to behave differently" and "it does." I'd write the logic and spec the behaviour; the tools helped me implement and debug faster than I could alone. The architecture, the decisions, and the mistakes are all mine.

The iteration loop was roughly weekly: run the screener, review candidates against real charts, find where the model disagreed with reality, trace the cause, and adjust. Most calibration work happened through a standalone diagnostic script that let me test individual tickers against the detection logic in isolation — which kept the main app stable while I was experimenting.

The hardest thing to figure out: the screener was surfacing setups with technically valid stop levels that were economically useless — the model was locking onto old tight bases from lower price levels, weeks or months after a large rally had made them irrelevant. Diagnosing it required building a 12-case backtest harness and tracing exactly which calculation was anchoring to the wrong reference point. The fix was architectural: changing how the pivot detector identifies swing points. It's the kind of issue that's invisible until you've tested it against real data across enough different scenarios.
05 — one concrete result
Calibration finding — April 2026
The breakout filter was the primary bottleneck — and the data made it obvious.
Early screener runs were returning almost no candidates. Systematic one-change-at-a-time testing revealed that a 5% breakout filter — originally intended to exclude stocks that had already moved — was eliminating the majority of viable setups before they ever reached scoring. A single threshold change to 10% tripled the candidate yield without degrading setup quality. That finding led to a broader recalibration pass: the RS percentile hard gate was restructured from a binary SPY comparison to a universe-ranked composite, the scoring weights were redistributed across five components, and three additional hard filters were added. None of that would have been diagnosable without the diagnostic tooling or the structured screener run history.
06 — what's next
  • Live calibration period
    Running the screener consistently for 4–6 weeks, logging all candidates and outcomes, and building enough real data to make confident structural changes rather than speculative ones.
  • Zigzag pivot detector
    Replacing the fixed rolling-window pivot detection with a percentage-based swing detector — the highest-leverage architectural improvement identified, deferred until live data confirms it's the right priority.
  • Fundamentals scoring layer
    Wiring in the FMP API for earnings growth, revenue growth, and return on equity — currently scaffolded in the scoring model but not yet live in screener runs due to data source limitations.
  • Order preparation module
    Structured order ticket generation — translating the pre-trade checklist output into a ready-to-review order spec, as a first step toward execution integration.
07 — why this matters for SE roles
Building this system taught me something I'd suspected but can now demonstrate: the hardest part of any tool isn't the technology, it's understanding the real workflow well enough to know what the tool actually needs to do. Every meaningful improvement in this project came from using it — running real screener outputs, tracing where the model disagreed with my own judgment, and working backwards to the cause. That cycle — deploy, observe, diagnose, improve — is exactly what good solutions engineering looks like in practice. I'm not looking to be handed a spec. I'm drawn to the part of the role where you figure out what the spec should have been.
